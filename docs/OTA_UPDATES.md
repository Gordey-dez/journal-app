# OTA (Over-The-Air) Обновления

## Текущая конфигурация

Проект настроен для OTA обновлений через Expo Updates:

- ✅ `expo-updates` установлен и настроен
- ✅ Runtime version: `appVersion` (привязан к версии в `app.json`)
- ✅ Каналы обновлений: `development`, `preview`, `production`
- ✅ Автоматическая проверка обновлений при запуске

## Что можно обновлять через OTA

### ✅ Можно обновлять (без пересборки)

1. **JavaScript/TypeScript код**
   - Все файлы в `app/`, `src/`, `components/`
   - React компоненты
   - Бизнес-логика
   - Утилиты и хелперы

2. **UI и стили**
   - Стили компонентов
   - Цвета и темы (если не меняется структура)
   - Layout и разметка

3. **Конфигурация роутинга**
   - Изменения в структуре экранов
   - Параметры навигации

4. **Тексты и строки**
   - Все текстовые строки в компонентах
   - Сообщения об ошибках
   - Подсказки и хинты

5. **Ассеты (частично)**
   - Изображения (если не меняются размеры критично)
   - Шрифты (если уже включены в билд)

### ❌ НЕЛЬЗЯ обновлять через OTA

1. **Нативные зависимости**
   - Новые npm пакеты с нативным кодом
   - Изменения версий нативных модулей
   - Новые Expo модули, требующие нативной конфигурации

2. **Версия Expo SDK**
   - Обновление Expo SDK требует нового билда

3. **Конфигурация приложения**
   - Изменения в `app.json` (частично)
   - Изменения в `eas.json`
   - Новые permissions
   - Изменения bundle identifier / package name

4. **Иконки и splash screen**
   - Требуют пересборки приложения

5. **Изменения в `package.json`**
   - Новые зависимости требуют нового билда

6. **Нативные плагины**
   - Новые плагины в `app.json` требуют пересборки

## Как работают обновления

### Runtime Version

В `app.json` настроено:
```json
"runtimeVersion": {
  "policy": "appVersion"
}
```

Это означает, что `runtimeVersion` = версия из `app.json` (сейчас "1.0.2").

**Важно:** Обновления OTA работают только для приложений с одинаковым `runtimeVersion`. Если вы измените версию в `app.json`, пользователям нужно будет обновить приложение через магазин.

### Каналы обновлений

В `eas.json` настроены каналы:
- `development` - для development builds
- `preview` - для preview builds
- `production` - для production builds

Каждый канал получает обновления независимо.

## Процесс публикации обновления

### 1. Внесите изменения в код

Измените любой JavaScript/TypeScript код, который можно обновить через OTA.

### 2. Опубликуйте обновление

```bash
# Для production канала
eas update --branch production --message "Описание изменений"

# Для preview канала
eas update --branch preview --message "Описание изменений"
```

### 3. Обновление автоматически распространится

- При следующем запуске приложения будет проверка обновлений
- Если найдено обновление, пользователь получит уведомление
- После подтверждения приложение обновится

## Автоматическая проверка обновлений

В проекте добавлен компонент `UpdateChecker`, который:
- Автоматически проверяет обновления при запуске приложения
- Показывает диалог пользователю при наличии обновления
- Применяет обновление после подтверждения

## Ручная проверка обновлений

Можно добавить кнопку в настройках для ручной проверки:

```tsx
import { useUpdateChecker } from "@/src/components/UpdateChecker";

function SettingsScreen() {
  const { checkAndUpdate, isChecking, canUpdate } = useUpdateChecker();

  return (
    <Pressable onPress={checkAndUpdate} disabled={!canUpdate || isChecking}>
      <Text>Проверить обновления</Text>
    </Pressable>
  );
}
```

## Ограничения

1. **Только production builds**: OTA обновления не работают в Expo Go и development режиме
2. **Runtime version**: Обновления работают только в рамках одного runtime version
3. **Размер обновления**: Рекомендуется не более 50MB на обновление
4. **Сеть**: Требуется интернет-соединение для загрузки обновлений

## Рекомендации

1. **Тестируйте обновления** на preview канале перед публикацией в production
2. **Версионируйте изменения** - используйте осмысленные сообщения в `eas update --message`
3. **Мониторьте обновления** через Expo Dashboard
4. **Резервные планы** - всегда имейте готовый новый билд на случай критических изменений

## Когда нужен новый билд

Создавайте новый билд, если:
- Добавили новую нативную зависимость
- Обновили Expo SDK
- Изменили иконки или splash screen
- Изменили bundle identifier / package name
- Добавили новые permissions
- Изменили runtime version

## Полезные команды

```bash
# Проверить текущую версию обновления
eas update:list

# Посмотреть информацию о проекте
eas project:info

# Откатить обновление (если нужно)
eas update:republish --branch production
```
